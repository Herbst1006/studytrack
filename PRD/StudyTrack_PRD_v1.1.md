## 一、文档说明

### 1.1 产品概述

StudyTrack 是一款面向**自主学习者**的学习记录工具。它不假设用户有明确的终点，而是帮助用户把每一次学习行为沉淀下来，让积累本身变得可见。

核心理念：**持续记录，而非追踪进度。**

大多数学习工具把「完成目标」作为衡量标准，但真实的学习往往没有固定终点——读一本书不一定有截止日，学一门语言没有满分线，探索一个领域更无法量化。强制填写目标反而会把大量真实用户挡在门外，并制造不必要的焦虑感。

StudyTrack 的起点是：**记录本身就有价值。** 当你能看见自己「在学 Python 这件事上已经投入了 31 小时」「费曼物理学已经读了 9 章」，这个积累的轨迹本身就是驱动力——不需要进度条告诉你还差多少。

对于确实有目标的用户（备考、课程完成等），进度条仍然可以选择开启。两种模式并存，用户自己决定。

本文档覆盖 MVP 阶段（v1.1）的完整需求，包含用户研究结论、竞品分析、功能规格与原型说明。

### 1.2 文档范围

- 目标读者：产品经理、开发工程师、设计师
- 本文档仅覆盖 Web 端 MVP，移动端为 V2 迭代计划
- 版本历史：v1.0（初稿）

## 二、用户研究

### 2.1 研究方法

本次用户研究采用**二手数据挖掘**方式，通过 Python 脚本调用 Apple iTunes RSS 官方接口采集竞品评论：

- 采集范围：Forest、番茄ToDo、滴答清单、Notion
- 原始数据：1355 条评论
- 分析样本：326 条低分评论（1-3星）
- 分析方法：人工阅读 + 主题归类

### 2.2 目标用户画像

**核心用户：自主学习者**

| 维度 | 描述 |
|------|------|
| 身份 | 在校大学生、自学转行者、考证备考者、职场技能提升者 |
| 核心特征 | 同时在推进 2-5 件学习事项；学习行为零散但持续；不一定有明确终点，但想看见自己的积累 |
| 典型场景 | 「我在学 Python、读费曼物理、练吉他，没有截止日，但想知道自己有没有在坚持」 |
| 现有解法 | 备忘录、Excel、番茄钟类 App 分开使用，工具碎片化，没有整体视图 |
| 决策动机 | 想看见积累的轨迹；希望记录行为本身产生驱动力；对数据丢失有强烈恐惧 |

> **定位说明：** 不限定身份，以「自主学习动机」圈定用户。不假设用户有明确目标，目标量为选填项——有目标的用户看进度条，没有目标的用户看累计轨迹，两种体验都完整。

### 2.3 核心用户洞察（5大痛点）

| # | 痛点主题 | 频次 | 典型用户原话 | 对产品的启示 |
|---|----------|------|--------------|--------------|
| 1 | 商业化过度 / 付费背刺 | ~265条 | 「买断制现在突然更新要我重新订阅，太恶心」 | 核心功能全部免费，定价模式从一开始讲清楚 |
| 2 | 数据丢失 / 账号崩溃 | ~95条 | 「连续种了五千多棵树，一更新数据全没了」 | 学习记录可导出，数据是用户资产不是平台筹码 |
| 3 | 功能臃肿 / 失去初心 | ~35条 | 「从自律工具变成集卡集邮，失去了意义」 | 核心流程不超过3步，功能保持克制 |
| 4 | 积累不可见 / 统计功能弱 | ~28条 | 「连自己的图表统计都要Plus才能看」 | 学习轨迹可视化是核心功能，免费提供；无论有无目标都能看见自己的积累 |
| 5 | 打卡机制容易失效 | ~15条 | 「断签就清零，谁来谁受气」 | 用活跃率替代连续天数，容错不惩罚 |

## 三、竞品分析

### 3.1 竞品选取

覆盖「专注类」「任务管理类」「时长追踪类」三个方向，共 5 款：

- **Forest 专注森林** — 专注类代表，种树打卡机制
- **番茄ToDo** — 番茄钟计时为核心
- **iHour** — 基于一万小时定律的时长追踪（最直接竞品）
- **滴答清单** — 任务管理类代表
- **Notion** — 通用笔记/任务工具

### 3.2 功能对比矩阵

| 功能维度 | Forest | 番茄ToDo | iHour | 滴答清单 | **StudyTrack** |
|----------|--------|----------|-------|----------|----------------|
| 多项目进度总览 | ✗ | ✗ | △ 仅时长 | ✗ | **✓ 免费提供** |
| 完成率/进度条 | ✗ | ✗ | △ 时长达成 | △ 付费 | **✓ 免费提供** |
| 周/月统计图 | △ 付费Plus | △ 付费 | ✓ | △ 付费 | **✓ 免费提供** |
| 容错打卡机制 | ✗ | ✗ | ✗ | ✗ | **✓ 活跃率不清零** |
| 数据导出 | ✗ | ✗ | △ 部分 | △ 付费 | **✓ 免费CSV（V2）** |
| 学习项目管理 | ✗ | △ 标签 | ✓ 项目分类 | △ 通用清单 | **✓ 专为学习设计** |
| 核心功能免费 | ✗ | △ 基础免费 | △ 基础免费 | ✗ | **✓ 全免费** |

> 注：✓ 支持 · △ 部分支持/付费 · ✗ 不支持

### 3.3 最直接竞品：iHour

iHour 基于「一万小时定律」，核心是记录单个技能的时间积累。与本产品的关键差异：

- iHour 回答「我在这件事上花了多久」，StudyTrack 回答「我所有学习项目现在进展如何」
- iHour 不支持多项目并行进度对比，不支持章节完成率
- StudyTrack 切入「多任务学习者的整体视图」，是 iHour 未覆盖的场景

### 3.4 差异化定位

> **让每一次学习行为都被看见——记录积累，而非追赶目标。核心功能完全免费。**

## 四、产品目标

### 4.1 MVP 目标

- 让用户记录每次学习行为后，能立刻在首页看见自己所有项目的积累状态
- 目标量设为选填：有目标的用户看进度条，无目标的用户看累计轨迹
- 提供「打卡 → 积累更新 → 可视化反馈」的完整闭环
- 核心功能 100% 免费，建立用户信任
- Web 端上线，完成可测试的 Demo

### 4.2 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| 打卡留存率 | 测试用户 7 日留存 ≥ 40% | 观察打卡记录数量 |
| 核心路径完成率 | 新用户添加项目并完成首次打卡 ≥ 70% | 用户测试观察 |
| 用户满意度 | 原型测试 SUS 评分 ≥ 70 | 问卷收集 |

## 五、功能需求

### 5.1 功能优先级总览

| 优先级 | 功能模块 | 开发难度 | 决策依据 |
|--------|----------|----------|----------|
| **P0** | 学习项目管理 | 低 | 数据载体，无此则其他功能无从挂载；竞品均未针对学习场景专项设计 |
| **P0** | 学习打卡记录 | 低 | 核心数据入口；用户明确诉求；操作需 ≤3 步 |
| **P0** | 多项目进度总览 | 中 | 最大竞品空白；直接解决「不知道学到哪了」的核心痛点 |
| **P0** | 学习时长统计图 | 中 | 竞品均锁付费墙；用户抱怨最集中的功能点之一 |
| **P0** | 容错打卡机制 | 低 | 所有竞品空白；低成本高价值差异点 |
| P1 | 学习热力图日历 | 中 | 视觉冲击力强，适合分享；V2 加入 |
| P1 | 目标截止日期提醒 | 低 | 增强目标感；MVP 阶段非必要 |
| P1 | 数据导出 CSV | 低 | 建立数据信任感；V2 加入 |
| P1 | 学习笔记备注 | 低 | 增加打卡内容深度；V2 加入 |
| 暂缓 | 社交排行 | 高 | 开发复杂；用户痛点数据不支持 |
| 暂缓 | 番茄钟计时器 | 中 | 竞品已饱和；与本产品定位不重叠 |
| 暂缓 | AI 学习计划生成 | 高 | 开发周期长；需求未充分验证 |
| 暂缓 | 移动端 App | 极高 | Web Demo 足够验证；App 开发周期是 Web 的3倍 |

### 5.2 P0 功能详细规格

#### F1 · 学习项目管理

| 字段 | 说明 |
|------|------|
| 功能描述 | 用户可创建、编辑、归档学习项目，每个项目代表一件正在学的事 |
| 必填字段 | 项目名称（最长30字）、记录方式（时长 or 章节数） |
| 选填字段 | 目标总量、标签颜色、截止日期 |
| 两种模式 | 设了目标量 → 显示进度条和完成百分比；未设目标量 → 显示累计数量和积累轨迹，两种模式体验同等完整 |
| 交互规格 | 首页展示状态卡片；支持拖拽排序；支持归档不删除；项目数量不限制 |
| 边界条件 | 无项目时展示空状态引导页；归档项目不计入活跃率统计 |

#### F2 · 学习打卡记录

| 字段 | 说明 |
|------|------|
| 功能描述 | 用户完成一次学习后，记录本次学习的项目、时长或章节数，自动累加进进度 |
| 核心操作路径 | 选择项目 → 输入本次进度（时长 or 章节数）→ 确认提交，共 **3 步** |
| 快捷入口 | 首页项目卡片右侧「+ 打卡」按钮，无需跳转页面 |
| 边界条件 | 同一天同一项目可多次打卡，累加计算；支持补录历史（选择日期）；时长最小单位 5 分钟 |

#### F3 · 学习状态总览

| 字段 | 说明 |
|------|------|
| 功能描述 | 首页以卡片列表展示所有进行中项目的当前状态，用户一屏获取全局视图 |
| 有目标项目 | 显示进度条、完成百分比、完成量/目标量、距截止日天数（若设置） |
| 无目标项目 | 显示累计数量、积累点阵（类似 GitHub contribution 风格）、最近打卡时间 |
| 排序逻辑 | 默认按「上次打卡时间」倒序排列；支持手动拖拽自定义顺序 |
| 状态标识 | 有目标时：进度 <30% 橙色警示，>80% 绿色高亮；无目标时不显示警示色 |
| 设计说明 | 两种模式并列展示，不做优劣区分——有目标和无目标的项目可以同时出现在首页 |

#### F4 · 学习时长统计图（免费）

| 字段 | 说明 |
|------|------|
| 功能描述 | 统计页展示学习时长的周/月趋势图，按项目分色堆叠展示，免费提供全部数据 |
| 图表类型 | 堆叠柱状图（按项目分色）/ 折线图（总时长趋势）/ 可切换周视图和月视图 |
| 数据范围 | 最近 4 周 / 最近 3 个月，全部历史数据免费可见 |
| 差异化说明 | Forest / 滴答清单将此功能锁在付费墙后，本产品免费提供，是明确的用户承诺 |

#### F5 · 容错打卡与积累感设计

| 字段 | 说明 |
|------|------|
| 核心设计 | 用「近30天活跃率」替代「连续打卡天数」作为主要激励指标，断签不清零 |
| 活跃率定义 | 近 30 天内有打卡记录的天数 / 30，展示为百分比（如「近30天 73% 活跃」） |
| 积累感设计 | 无论有无目标，每次打卡后的数字都只增不减——累计时长/章节数是用户的「学习资产」，永久保留 |
| 保留连续天数 | 连续打卡天数作为次要信息展示，不作为主要激励指标，不清零不惩罚 |
| 设计依据 | 「断签就清零，谁来谁受气」——来自真实用户评论；积累感的本质是：让用户感受到自己在变多，而非距离目标还差多少 |

## 六、信息架构与页面清单

### 6.1 核心页面

| 页面 | 主要功能 | 核心元素 |
|------|----------|----------|
| 首页（总览） | 项目进度一览 | 项目卡片列表 / 总体活跃率 / 快捷打卡入口 / 今日打卡状态 |
| 打卡页 | 记录单次学习 | 选择项目 / 输入时长或章节 / 日期选择 / 备注（V2） |
| 项目详情页 | 单项目深度数据 | 进度详情 / 打卡历史记录 / 编辑目标 |
| 统计页 | 趋势与汇总 | 时长趋势图 / 项目分布 / 活跃率热力图（V2） |
| 设置页 | 基础配置 | 项目管理 / 数据导出（V2）/ 关于 |

### 6.2 核心用户流程

**新用户流程（首次使用）**

1. 进入首页 → 看到空状态引导 → 点击「添加第一个项目」
2. 填写项目名称、目标类型、目标量 → 保存
3. 返回首页看到项目卡片 → 点击「+ 打卡」
4. 填写本次学习时长 → 提交 → 进度条更新 → 完成首次完整体验

**日常使用流程（回访用户）**

1. 打开应用 → 首页一屏看到所有项目进度
2. 选择当日完成的项目 → 打卡记录 → 进度实时更新
3. 周末查看统计页 → 了解本周各项目学习分布

## 七、非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 首页加载时间 <1.5s；打卡提交响应 <500ms |
| 数据安全 | 用户数据本地存储（MVP 阶段）；明确告知用户数据存储位置；V2 支持云同步 |
| 兼容性 | 支持 Chrome / Safari / Edge 最新版；响应式布局适配 1280px 以上屏幕 |
| 可用性 | 新用户无需注册即可使用（MVP 阶段本地数据）；核心操作路径不超过 3 步 |

## 九、附录

### A. 数据来源说明

- 竞品评论数据：通过 Python 脚本调用 Apple iTunes RSS 官方接口采集
- 采集数量：1355 条原始评论，326 条低分评论（1-3星）纳入分析

### B. 关键名词定义

- **学习项目**：用户自定义的一个学习目标单元，可以是课程、技能、考试备考或任何学习内容
- **打卡**：用户完成一次学习后的进度记录行为，包含时长或章节数
- **活跃率**：近30天内有打卡记录的天数占30天的比例，用于衡量学习坚持度
- **P0 功能**：MVP 必须上线的核心功能，缺少任意一项则产品无法成立

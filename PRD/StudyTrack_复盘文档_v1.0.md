### 决策 1：目标用户从「在校学生」扩展为「自主学习者」

**背景：** 项目最初定位是「面向在校学生的学习管理工具」。

**转变原因：**
- 「学了很多但不知道到哪了」这个痛点与身份无关，自学转行者、考证备考者同样有
- 只做学生市场规模有限，毕业后用户流失
- 以「学习动机」而非「身份」圈人，覆盖面翻倍，开发量不变

### 决策 2：从「进度追踪」转向「持续记录」

**背景：** v1.0 PRD 的核心概念是「帮用户看清所有项目的整体进度」，目标量为必填项。

**触发转变的问题：** 「有的学习者不清楚自己到底要学多少，进度条那块感觉得换掉」——这句话指出了一个真实的产品假设错误：不是所有学习行为都有明确终点。

**转变内容：**
- 产品理念：从「追赶目标」转向「记录积累，让积累本身可见」
- 功能规格：目标量改为选填；新增「无目标模式」（显示累计轨迹而非进度条）
- 差异化定位重写：「让每一次学习行为都被看见」

**类比参照：** GitHub contribution graph——从不规定你今天要 commit 几次，但那张热力图会让你自己想保持节奏。StudyTrack 想做学习领域的这个东西。

### 决策 3：打卡机制用「活跃率」替代「连续天数」

**背景：** 竞品评论中有明确抱怨：「断签就清零，谁来谁受气」。

**设计逻辑：** 「近30天活跃率」= 有打卡天数 / 30，断签不清零，关注趋势而非连续性。连续天数保留但作为次要信息展示。

**权衡：** 连续打卡天数有明确的激励感（「我已经坚持了30天」），活跃率的激励感相对弱一些，需要通过设计补足。

**现在怎么看：**
<!-- 活跃率这个指标用户真的能直觉理解吗？「73% 活跃」比「连续21天」更有激励感吗？有没有更好的方案？ -->
待补充

### 决策 4：暂不做番茄钟计时器

**背景：** 最初功能清单里有讨论是否加入番茄钟。

**决策理由：**
- Forest、番茄 ToDo 在这个功能上已经做得非常成熟，跟进没有差异化
- StudyTrack 管「之后看到什么」（积累轨迹），不管「当下怎么专注」（实时计时）
- 两个场景用户完全可以分开用，不需要做成一个产品

**现在怎么看：**

## 四、竞品调研洞察

### 最重要的一个发现
竞品（特别是 Forest）的大量差评集中在商业模式问题（买断转订阅、功能被付费墙拦截），而不是核心功能设计问题。这说明：**用户对「学习记录」类产品的核心功能诉求实际上没有被好好满足过**，大家的精力都花在骂商业化，功能层面的改进空间反而没人去做。

### iHour 是最直接竞品，但切入点不同
- iHour 问：「你在这件事上花了多久」（单项目，时长积累）
- StudyTrack 问：「你现在学的这些事情，整体状态是什么样的」（多项目，积累可视化）

### Web Demo 开发复盘

**遇到的 Bug 和解法：**

| Bug | 原因 | 解法 |
|-----|------|------|
| 打卡/添加弹窗一点就关闭 | `onclick` 事件冒泡到 overlay | 改用 `addEventListener` + `stopPropagation()` |
| 本月统计无数据 | 本月数据逻辑未写完 | 单独维护 `monthData` 对象，切换 seg 时重新渲染 |
| 进度条强制要求目标量 | 最初设计默认有目标 | 目标量改为 `null`，渲染逻辑分两路 |

**如果要继续迭代，下一步做什么：**
- 数据持久化（localStorage 或简单后端）
- 热力图日历（V2 核心功能）
- 响应式适配移动端
